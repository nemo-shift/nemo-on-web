# 초정밀 모듈화 및 엔진 일원화 트러블슈팅 보고서 (2026-03-22)

---

## 1. 배경: "비대해진 엔진의 한계와 디테일의 반란"
V4.3의 **Strict Whole-Pin** 아키텍처는 견고했으나, 기능 추가에 따라 `GlobalInteractionStage.tsx`가 600줄에 육박하는 거대 모듈이 되었습니다. 이로 인해 1px 단위의 정밀 제어가 어려워졌고, 코드 변경 시마다 예측 불가능한 사이드 이펙트가 발생하기 시작했습니다.

## 2. 엔진 숙청: "프레이머 모션(Framer Motion) 제거"
### ❌ 공존의 실패
- **발전 단계**: 프로젝트 초기에는 가벼운 컴포넌트 애니메이션에 Framer Motion을 혼용했습니다.
- **문제점**: 두 개의 애니메이션 엔진이 CPU 점유율을 나누어 가지면서 마스터 타임라인과 하위 애니메이션 사이에 **'미세한 엇박자'**가 발생했습니다. 특히 `RotatingText` 같은 고주파 애니메이션에서 프레임 드랍이 두드러졌습니다.
- **결단**: "네모:ON의 심장은 하나여야 한다." Framer Motion을 완전히 도려내고 모든 하위 요소를 **GSAP(`useGSAP`)**으로 재구현했습니다. 결과적으로 완벽한 '프레임 동기화'와 번들 사이즈 축소를 동시에 달성했습니다.

## 3. 아키텍처 진화: "빌더 패턴(`builders.ts`)의 도입"
### 🏗️ 건물 짓기에서 조각하기로
- **문제**: 타임라인 로직이 컴포넌트 내부에 직접 기술되어 있어, 가독성이 떨어지고 수정 시 전체 무결성을 확인하기 어려웠습니다.
- **해결**: 애니메이션 생성을 전담하는 **`builders.ts`**를 분리했습니다. 컴포넌트는 이제 '명령'만 내리고, 빌더는 '시나리오'를 조립합니다.
- **성과**: 인터랙션이 이제 하드코딩된 코드가 아닌, 수정 가능한 '스크립트'처럼 관리되기 시작했습니다.

## 4. 디테일 영점 조절 (Core Fixes)
### 4.1 푸터 리빌의 유령 노출 (Ghosting Footer)
- **증상**: 스크롤 바닥 도달 전 푸터가 미리 보이는 현상.
- **해결**: `REVEAL_FOOTER` 전용 가중치(0.25)를 할당하고 `totalWeight` 공식을 보정하여 스크롤 100%와 노출 100%를 수학적으로 일치시켰습니다.

### 4.2 로고 전이의 퀀텀 점프 (Scale Jitter)
- **증상**: 빅 로고가 헤더 로고로 변할 때 크기가 순간적으로 '점프'하는 현상.
- **해결**: CSS `clamp()` 가변 폰트와 GSAP `fromTo` 간의 렌더링 충돌을 방지하기 위해 뷰포트별 **공통 정규화 헬퍼**를 도입했습니다.

## 5. 결론 및 유산
V12.0 모듈화의 핵심은 **"복잡함을 수용하는 것이 아니라, 복잡함을 잘 쪼개는 것"**이었습니다. 이제 프로젝트는 어떤 대규모 콘텐츠가 들어와도 builders.ts를 통해 질서 있게 수용할 준비가 되었습니다.
